# Copilot Operating Instruction — Outcome Dealer OS

You are working only in the GitHub repo `dl-blackline/outcome-dealer-os`.

Do not touch Power Prospect. Do not reference Power Prospect as the active build target.

This repo is **Outcome Dealer OS**, a premium AI-native dealership operating system. The current repo has a strong product vision and some shell and role foundations, but it is still partly scaffolded and still uses mock-driven surfaces in places.

Your job is to finish the repo carefully using the existing codebase as the chassis.

## Core rules

1. Do not build disconnected features.
2. Do not create duplicate business objects if canonical ones already exist.
3. Preserve and elevate the workstation as a first-class execution layer.
4. Replace placeholder shell-only behavior with real route and page structure.
5. Keep approvals, audit, and events visible and coherent in the product.
6. Keep the UI premium, dark-first, restrained, and enterprise-grade.
7. Prefer targeted refactors over reckless rewrites.
8. Update docs whenever architectural reality changes.
9. Preserve what is already good: role model, shell direction, north-star architecture.
10. Explicitly separate implemented, placeholder, and planned functionality.

## Working method

- Inspect existing code before changing it.
- Explain what you are preserving.
- Explain what you are changing.
- Implement the files accordingly.
- Summarize assumptions, risks, and next steps.

## Priorities

- route-driven app shell
- workstation integration
- real record and ops pages
- canonical frontend data layer
- approvals, audit, and event continuity
- repo stabilization for final review

## Avoid

- flashy AI gimmicks
- disconnected placeholder modules
- buried workstation navigation
- Spark-template leftovers where they confuse product identity
- duplicate role, permission, or event definitions

Build Outcome Dealer OS as one coherent operating system.
